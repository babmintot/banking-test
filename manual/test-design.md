# Тест-дизайн ручного тестирования

---

## 1. Цель тестирования

Целью тестирования является проверка корректности работы веб-сервиса перевода денежных средств с банковского счёта на карту.

В рамках тестирования необходимо убедиться в корректности:
- выбора счёта для списания;
- ввода и валидации номера банковской карты;
- ввода и обработки суммы перевода;
- расчёта комиссии;
- определения доступности операции перевода;
- отображения сообщений об ошибках.

---

## 2. Объект тестирования

Веб-приложение, реализующее функциональность перевода денежных средств с пользовательского счёта на банковскую карту.

Тестирование проводится через пользовательский интерфейс (UI).

---

## 3. Основание для тестирования

Требования к системе (согласно заданию):

- номер карты должен состоять из **ровно 16 цифр**;
- доступная сумма рассчитывается по формуле:

  доступная сумма = баланс − резерв

- комиссия составляет **10% от суммы перевода** и округляется вниз;
- перевод возможен, если:

  сумма перевода + комиссия ≤ доступная сумма

- при превышении доступного остатка должно отображаться сообщение:

  Недостаточно средств на счёте

---

## 4. Подход к тестированию

Тестирование проводится вручную с использованием браузера.

Основной подход:
- функциональное тестирование пользовательского интерфейса;
- проверка бизнес-логики приложения;
- проверка валидации пользовательского ввода.

---

## 5. Тестовые техники

При проектировании тестов использованы следующие техники тест-дизайна:

- **Позитивное тестирование**  
  Проверка корректной работы системы при валидных данных.

- **Негативное тестирование**  
  Проверка реакции системы на некорректный ввод.

- **Граничные значения (Boundary Value Analysis)**  
  Проверка поведения системы на границе допустимых значений.

- **Классы эквивалентности**  
  Разделение входных данных на корректные и некорректные группы.

---

## 6. Область покрытия

В рамках тестирования проверяются следующие функциональные области:

### 6.1 Работа с картой
- ввод номера карты;
- проверка длины номера карты (меньше 16, ровно 16, больше 16 символов).

### 6.2 Работа с суммой перевода
- ввод корректной суммы;
- ввод некорректной суммы (отрицательная, пустая);
- проверка граничных значений.

### 6.3 Бизнес-логика
- расчёт комиссии;
- проверка доступности перевода;
- корректность сравнения суммы и доступного остатка.

### 6.4 Пользовательский интерфейс
- отображение кнопки перевода;
- отображение сообщений об ошибках;
- динамическое появление элементов формы.

---

## 7. Покрываемые сценарии

В ходе тестирования реализованы следующие сценарии:

1. Успешный перевод при достаточном остатке;
2. Отказ при недостатке средств;
3. Проверка граничного значения суммы перевода;
4. Проверка корректной длины номера карты;
5. Проверка некорректной длины номера карты;
6. Проверка отрицательной суммы;
7. Проверка пустого значения суммы;
8. Проверка корректности расчёта комиссии.

---

## 8. Тестовые данные

Для тестирования используется следующий URL:

http://localhost:8000/?balance=30000&reserved=20001

Расчёт:
доступная сумма = 30000 − 20001 = 9999

Пример граничного значения:
сумма = 9099  
комиссия ≈ 900  
итого ≈ 9999 (граница)

---

## 9. Ожидаемые результаты

- корректная обработка валидных данных;
- корректная блокировка операций при нарушении условий;
- отображение ошибок при некорректном вводе;
- корректный расчёт комиссии;
- корректная работа системы на граничных значениях.
